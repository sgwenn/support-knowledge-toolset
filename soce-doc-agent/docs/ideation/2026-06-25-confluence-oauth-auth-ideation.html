Grounding Context

Project

soce-doc-agent — Python/FastAPI app that auto-generates Confluence KB articles from Jira tickets using Claude AI. Single-user local tool. Hackathon context — favor shippable, demo-ready improvements.

Current Auth State

  • Tokens stored in plaintext .env + os.environ
  • Bootstrap: one-shot CLI script, prints to stdout, manual copy-paste
  • Refresh fires reactively on 401 only — no expiry tracking
  • On refresh failure: unhandled RuntimeError → HTTP 500
  • Zero UI auth surface — no button, no route, no status indicator
  • Auth errors surface as generic "Server error (HTTP 401) — check the server logs"

Key Code Seams

  • _get_token() at confluence.py:108
  • _persist_env_tokens() at confluence.py:41
  • _refresh_token() at confluence.py:71–105
  • classifyError() at app.js:14–31
  • db.py — existing SQLite connection

External Constraints

  • Atlassian rotating refresh tokens — each refresh invalidates the previous token immediately
  • Crash between refresh API call and write = permanent access loss
  • offline_access scope required for any refresh token
  • Access tokens expire in ~3600s; refresh tokens ~365 days

1.

In-App OAuth Routes: /auth/confluence + /auth/callback

auth-trigger Confidence 95% Complexity: Medium
Description

Add two FastAPI routes to main.py: GET /auth/confluence redirects the browser to the Atlassian consent screen, and GET /auth/callback receives the authorization code, exchanges it for tokens, and writes them to SQLite (see Idea 2). The callback URL registered in the Atlassian developer console becomes the app's own URL — no separate process, no port juggling. The existing scripts/atlassian_oauth.py logic (already a working OAuth handler) becomes dead code. A "Connect Confluence" button in the UI opens /auth/confluence in a new tab; on success the tab closes and the status pill (Idea 4) turns green.

This is exactly what the user asked for: a button click that opens the OAuth flow, then returns with a working connection.

Before: 5 manual steps across terminal, browser, text editor, server. After: 3 clicks, fully in-browser.

Basis

direct: scripts/atlassian_oauth.py:56 — runs HTTPServer(("localhost", 9876), CallbackHandler) as a completely separate process. main.py:74–184 — no /auth routes exist. external: Starlette route handles OAuth callbacks natively; same pattern as any FastAPI OAuth integration.

Rationale
Every new install or demo starts cold. The current path requires CLI access, copy-paste, and a server restart — five failure modes before the app is usable. Moving the callback into FastAPI collapses this to a single browser flow. This is the linchpin: without an in-app callback, the status indicator (Idea 4) has nowhere to redirect, and the reconnect CTA (Idea 5) has no target.
Downsides
Requires registering the in-app callback URL in the Atlassian developer console (one-time setup). Adds ~60 lines to main.py. state parameter for CSRF protection needs session storage (Starlette's SessionMiddleware handles this).
2.

SQLite Authoritative Token Store

token-storage Confidence 95% Complexity: Medium
Description

Replace _get_token() (confluence.py:108) and _persist_env_tokens() (confluence.py:41) — the two narrow seams already identified — with reads/writes to an oauth_tokens table in the existing SQLite DB (db.py). Schema: (key TEXT PRIMARY KEY, value TEXT, updated_at INTEGER). The .env file keeps only static config (CLIENT_ID, CLIENT_SECRET) — never rotating bearer credentials.

This is the foundation idea. Idea 1's callback writes here. Idea 3's expiry check reads here. Idea 6's idempotent COALESCE runs here. All three are one-liners once this seam exists.

Basis

direct: _get_token() at confluence.py:108 and _persist_env_tokens() at confluence.py:41 are confirmed as the only two storage write/read sites. db.py exists with a working SQLite connection (check_same_thread=False). external: SQLite Token table is the standard minimal-correct solution for single-user local tools; named explicitly in research grounding.

Rationale
The current .env write-back is the crash-danger zone. With Atlassian's rotating refresh tokens, a kill between os.environ mutation and os.replace invalidates the old refresh token and loses the new one — permanent lockout requiring a full re-auth. SQLite's BEGIN/COMMIT makes the write atomic. It also enables expiry tracking (expires_at) and an observable health endpoint — neither is possible with the .env model.
Downsides
The SQLite file is a new secret-bearing artifact that must be excluded from version control (.gitignore). Migration path from existing .env tokens needs one-time import on first run. ~40 lines of DB migration code.
3.

Proactive Pre-Refresh with expires_at Tracking

token-lifecycle Confidence 90% Complexity: Low–Medium
Description

Store expires_at = now + expires_in - 60 in the SQLite token table at every write (Atlassian returns expires_in in the token exchange response). Update _get_token() to check expires_at before returning: if within 60 seconds, call _refresh_token() proactively before serving the token. Add a FastAPI lifespan startup task that does the same check on boot.

The reactive 401-retry path stays as a last-resort safety net. In practice, the token is always fresh before any request.

Basis

direct: evidence-token-lifecycle.md §7 — "No expiry tracking — purely reactive, no proactive pre-refresh." confluence.py:71–105 — refresh fires only on 401. Atlassian token exchange response includes expires_in (confirmed in Atlassian docs). external: Standard proactive-refresh pattern; named in web research grounding.

Rationale
Atlassian access tokens expire in 3600s. A job that starts with a 59-minute-old token will 401 mid-generation — exactly when Claude is writing the KB article. Proactive refresh moves the cost to idle time and removes this failure mode from the critical path entirely. It also makes /auth/status (Idea 4) trivially implementable as expires_at > now.
Downsides
The 60-second pre-refresh window is a heuristic. If the access token is used under 60 seconds left, the old token may still be served for one request (then 401 falls back to reactive). This is acceptable — the 1-hour lifetime means this edge case is rare.
4.

Connection Status Indicator + /auth/status Endpoint

auth-ux Confidence 90% Complexity: Low
Description

Add GET /auth/status returning {"confluence": "ok" | "expired" | "unconfigured", "expires_in_seconds": N} backed by the SQLite expires_at field. On page load, app.js calls this once and renders a status pill in the header: green "Confluence: connected" or red "Confluence: not connected — click to connect." Clicking the red pill navigates to /auth/confluence (Idea 1).

Before
  • Click Generate
  • Wait 30–60 seconds
  • "Server error (HTTP 401) — check the server logs"
  • Open terminal, check logs, re-run CLI script
After
  • Open app → red pill visible instantly
  • Click pill → OAuth consent (Idea 1)
  • Pill turns green
  • Click Generate → works
Basis

direct: evidence-auth-ux.md confirms no auth status indicator exists anywhere in index.html or app.js. No /auth/status route in main.py. classifyError at app.js:14–31 has no Confluence-specific case — 401s surface as generic "Server error."

Rationale
Every Confluence 404 in the current app is invisible until publish. A status pill makes auth state legible in 200ms on load. For a hackathon demo, a visible "Confluence: connected ✓" indicator is the most legible signal that the system is ready — no explanation needed. It also composes with Idea 5: when a typed auth_error SSE event fires, it flips the pill red automatically.
Downsides
Polling /auth/status on every page load adds one HTTP round-trip. The endpoint must be unauthenticated (or use the session cookie from Idea 1) to avoid chicken-and-egg auth problems. For a local tool, this is acceptable.
5.

Structured Auth Error SSE Events + Reconnect CTA

auth-ux Confidence 85% Complexity: Low
Description

Add a typed auth_error SSE event emitted when a Confluence auth failure occurs: {"type": "auth_error", "service": "confluence", "action": "reconnect"}. In classifyError() at app.js:14–31, add a branch for type === "auth_error" that replaces the generic error banner with: "Confluence authentication expired — Reconnect to Atlassian" where the link navigates to /auth/confluence.

On the backend, normalize the three current silent failure patterns (None return, error-dict, bare RuntimeError) in confluence.py into a single ConfluenceAuthError that the SSE layer translates to the structured event.

Basis

direct: app.js:14–31 — classifyError has no Confluence-auth case; 401 shows as generic "Server error (HTTP 401) — check the server logs." agent.py:465–468 — catches httpx.HTTPStatusError and emits {"type": "error", "message": "Confluence API error 401"} with no remediation. Three distinct silent failure paths in confluence.py all produce different (or no) signals.

Rationale
Error specificity is a force multiplier on UX. One new SSE event type makes auth failures self-healing: the user sees "click here to reconnect" instead of "check the server logs." It also creates a shared signal channel — when Idea 3's proactive refresh fails, the same auth_error event fires and flips the status pill (Idea 4) red.
Downsides
Requires normalizing three existing failure patterns into one exception type — mechanical but necessary refactor across get_ts_space_id, search_confluence, and create_draft_page_pat. Risk of slightly different error semantics.
6.

Idempotent COALESCE Token Write

token-lifecycle Confidence 90% Complexity: Low
Description

When Atlassian omits the refresh_token field in a rotation response, the current code skips updating the value in memory (confluence.py:101–102: if new_refresh: os.environ[...] = new_refresh) — but _persist_env_tokens() is still called, potentially writing an empty string to .env. The SQLite fix is one SQL expression:

UPDATE oauth_tokens SET refresh_token = COALESCE(NULLIF(?, ''), refresh_token) WHERE key = 'atlassian'

The old refresh token is atomically preserved when the new value is empty. Combined with Idea 2's SQLite store, this correctness invariant is enforced at the data layer — impossible to bypass.

Basis

direct: confluence.py:101–102 — if new_refresh: os.environ["ATLASSIAN_REFRESH_TOKEN"] = new_refresh. An empty string from Atlassian means the condition is falsy; the in-memory value is preserved but the .env write still runs with whatever is in the dict, which may write an empty string. external: Atlassian rotating refresh tokens — a lost refresh token requires full re-authorization. This is the last correctness bug before the token store is reliable.

Rationale
This is a silent correctness bug that only manifests as a complete auth lockout requiring the full CLI re-auth cycle. One SQL expression fixes it permanently. It is the difference between "we might lose the refresh token" and "we provably cannot lose the refresh token."
Downsides
Depends on Idea 2 (SQLite store). As a standalone change to the .env model it would require extra conditional logic. Best implemented together with Idea 2 as a single migration.
7.

Move KB_API_KEY from URL Query Param to Authorization Header

security-posture Confidence 90% Complexity: Low
Description

The SSE stream currently connects as /stream/{job_id}?key=<KB_API_KEY> — the key appears in server access logs, browser history, and any proxy logs. app.js already sends X-Api-Key as a header for all non-SSE requests via apiHeaders(). The fix: replace the native EventSource with a fetch-based SSE polyfill that supports custom headers, then read request.headers.get("Authorization") in the stream route instead of request.query_params.get("key").

Basis

direct: main.py:110–111 — stream route reads key from request.query_params. app.js:35–37 — SSE connects with ?key=${encodeURIComponent(key)}. app.js:6–10 — apiHeaders() already uses X-Api-Key header for non-SSE calls. Native EventSource does not support custom headers; a fetch-based SSE polyfill (e.g., @microsoft/fetch-event-source or a 50-line inline equivalent) is the standard fix.

Rationale
A key in a URL is logged by every web server, proxy, and CDN by default. For a local tool this is low severity, but for a hackathon demo where judges inspect DevTools, a visible API key in the URL is an embarrassing finding. The fix is ~30 lines of JS and 2 lines of Python. It also unblocks moving to cookie-based auth for the Confluence OAuth session (the BFF pattern) as a future step.
Downsides
Replacing native EventSource adds a small fetch-SSE polyfill dependency (or ~50 lines of inline code). Reconnection logic (native SSE auto-reconnects; fetch-based needs explicit handling) needs care. Low risk — the existing 30-second stall detection in app.js handles this already.
