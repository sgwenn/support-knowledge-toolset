KB Agent Quality & Output

Date: 2026-06-24 Focus: Agent quality & output Topic: agent-quality-output Mode: Repo-grounded Survivors: 11 of ~30 candidates (updated 2026-06-24)

Codebase Context

soce-doc-agent is a FastAPI + SSE backend that runs an Anthropic API multi-step agentic loop (Claude Sonnet 4.6) to generate Confluence KB articles from Zendesk support tickets. The agent fetches ticket data from Snowflake, retrieves similar past tickets via vector cosine similarity, searches Confluence for existing articles, and produces a structured HTML draft for publish.

Confirmed Gaps

  • Jira lookup is a prompt instruction only — no code-level enforcement
  • similarity_score computed but never used downstream
  • SATISFACTION_RATING_SCORE fetched, never used in ranking
  • FULL_CONVERSATION can be NULL — fails silently
  • novelty_check exists but not called before publish
  • New articles publish to personal space, not SUPPORT space
  • EXISTING_URL search targets SUPPORT space — inconsistency
  • _parse_draft() validates string markers only, no section structure
  • coverage_records has no quality signal column
  • Publish appends with <hr/> on update — no merge logic

Key Files

  • prompts.py — SYSTEM_PROMPT (9-step workflow)
  • tools/snowflake.py — fetch_ticket, find_similar_tickets, novelty_check
  • main.py — FastAPI routes, SSE streaming
  • db.py — SQLite coverage_records schema
  • static/app.js — frontend event handling, publish flow

#1 R1 Retrieval quality

Hard-wire Jira escalation as deterministic pre-flight

Confidence 90% Complexity Medium Convergence 4 frames
Description
Move Jira lookup from a SYSTEM_PROMPT workflow instruction to a Python pre-flight step in run_kb_agent. When IS_ESCALATED_TO_JIRA_ESCALATION=True in the fetch_ticket result, fetch the Jira ticket via the Atlassian MCP before the agent loop starts and inject it as guaranteed structured context in the first user message. The agent loop receives Jira data unconditionally — it cannot skip it.
Jira payload moves from an optional agent instruction to a guaranteed Python pre-flight step.
Basis
direct agent.py:102 — messages list is built fresh each call with no guaranteed Jira context; prompts.py workflow step 2 instructs Jira lookup as an agentic step only; IS_ESCALATED_TO_JIRA_ESCALATION is returned in fetch_ticket result (snowflake.py:45) but there is no code-level enforcement that the agent calls Jira before generating the draft.
Rationale
Agentic loops are stochastic. A model pass that is confident from ticket metadata alone may skip the Jira lookup entirely with no code-level catch. Jira data is the highest-signal source — it carries the engineering root cause, often the only accurate explanation of why something failed. Pre-flight moves this signal into guaranteed context rather than hoping the model chooses to fetch it. Synergy: once Jira is deterministically present, source composition is known before the loop starts, enabling automatic confidence tier computation (see idea 5, P1).
Downsides
Pre-flight Jira fetch adds ~1–2s latency for all escalated tickets. Jira key may be stale or absent on some tickets (need a graceful skip). Implementation requires exposing the Atlassian MCP call as a standalone Python function callable outside the agent loop.
#2 A1 Article structure

Mandatory named section schema enforced at prompt + parse time

Confidence 85% Complexity Low Convergence 5 frames
Description
Define 5 required named sections as mandatory <h2> anchors in the SYSTEM_PROMPT output format: Symptoms, Root Cause, Resolution Steps, Verification, and Related Issues. Validate their presence in _parse_draft() and emit an error if required sections are missing. The agent fills a defined scaffold rather than producing free-form HTML.
Required h2 anchors create a consistent, scannable article shape across all generated articles.
Basis
direct _parse_draft() (agent.py:182–200) validates only string markers (DRAFT_TITLE:, EXISTING_URL:, DRAFT_HTML:) — no section content structure is checked. The SYSTEM_PROMPT specifies no required article sections. Every ideation frame independently identified the absence of section structure as the single highest-impact fix for article consistency.
Rationale
5-frame convergence — the strongest signal in this run. Dual enforcement at prompt (the agent knows the contract) plus parse time (code rejects violations) creates a hard quality gate rather than a soft instruction. This is foundational: it unlocks per-section inline editing (H1 combo) and section-targeted regeneration (H2 combo). Anatomy of a support KB article is well-understood; there is no reason for the structure to vary across tickets.
Downsides
Requires prompt tuning to avoid increasing generation failures — the model needs clear per-section guidance to reliably produce all 5 anchors. Some tickets may have insufficient data to populate every section meaningfully (e.g., a ticket missing a clear resolution will produce a thin Resolution Steps section).
#3 L1 Post-publish lifecycle

Publish to SUPPORT space as ai-generated draft

Confidence 91% Complexity Low
Description
Remove the personal-space lookup in run_publish_agent for new articles. Publish all new articles directly to the SUPPORT Confluence space with status: "draft" and an ai-generated label. This aligns the publish destination with the existing duplicate-check logic, which already targets SUPPORT space (SYSTEM_PROMPT step 5).
Basis
direct agent.py:262–297 — new articles use a multi-step personal-space lookup via Atlassian user account ID. SYSTEM_PROMPT step 5 instructs searching SUPPORT space for existing articles. These two are directly inconsistent: the agent deduplicates against SUPPORT space articles but publishes to a personal space where SUPPORT space dedup can never match its own output.
Rationale
Articles in personal spaces are invisible to other TEEs and unreachable by the EXISTING_URL search that determines whether an article already exists. The pipeline's dedup check can never match articles it has published — so every re-run on a related ticket creates a duplicate in some other personal space. Fixing the destination removes the inconsistency in roughly 15 lines of code in run_publish_agent. The highest-confidence fix in this candidate set.
Downsides
Requires SUPPORT space write permissions for the service account. Articles land in a shared space rather than a personal draft folder — the status: "draft" flag and ai-generated label are essential to prevent premature visibility. Should ship together with the pre-publish dedup gate (L2, cut from top 7) to prevent volume flooding in SUPPORT space.
#4 R6 Retrieval quality

Conversation quality banner

Confidence 92% Complexity Low
Description
Detect FULL_CONVERSATION IS NULL at fetch_ticket time and return a conversation_quality flag in the fetch result. Propagate the flag through the SSE done event to the frontend and render an amber warning banner at the top of the preview panel when the flag is set: "Draft generated from ticket summary fields only — full conversation was unavailable. Verify claims before publishing."
Amber banner in the preview panel when FULL_CONVERSATION is NULL — gives the TEE a trust signal before publishing.
Basis
direct snowflake.py:56–57 — FULL_CONVERSATION uses a LEFT JOIN to FACT_ZENDESK_TICKET_CONVERSATIONS_TECH_SUPPORT_AI, so it can be NULL for any ticket not yet in that table. prompts.py handles this fallback silently in instructions. No flag is returned to the frontend; no UI indication exists.
Rationale
When FULL_CONVERSATION is NULL, the article is synthesized from ticket metadata and pre-computed summary fields only — lower-quality, potentially incomplete inputs. A TEE cannot tell from the current UI whether their draft has full conversation context or not. The banner costs ~5 lines of Python and ~10 lines of JS and provides a meaningful trust signal at the moment it matters most.
Downsides
Banner wording must be informative without being alarming — a low-quality banner that fires frequently becomes noise. Should include the specific reason (no conversation data available yet) rather than a generic warning.
#5 P1 Prompt & synthesis

Source-weighted confidence tiers → Confluence callout macros

Confidence 78% Complexity Medium Convergence 3 frames
Description
Compute a deterministic confidence tier (HIGH / MEDIUM / LOW) from source composition at the end of the agent loop — before publish. Inject the tier instruction into the agent prompt so the generated article's sections carry Confluence info/warning/note macros calibrated to available sources. Alternatively, append the macro in Python post-parse based on source composition logged through the existing queue events.
Three deterministic tiers computed from which data sources were non-NULL at fetch time, rendered as Confluence callout macros visible in the published article.
Basis
direct prompts.py line ~39 says "if root cause not definitively confirmed, say so" — no format is specified and the instruction is model-dependent. Source composition (which data sources were non-NULL) is knowable in Python from the SSE queue events already flowing through run_kb_agent. Confluence ac:info/ac:note/ac:warning macros are valid Confluence storage format HTML.
Rationale
Confidence varies substantially based on input data quality. An article synthesized from a Jira engineering escalation is reliably more complete than one from ticket metadata alone. Encoding this as a visible macro in the Confluence article lets any TSE or reviewer immediately gauge trust — and the tier is deterministic (computable in Python), not dependent on the model's self-assessment. Synergy with R1: once Jira is pre-flighted deterministically, source composition is known before the loop starts.
Downsides
Tier boundaries require calibration — what counts as "Jira payload present" vs. "summary fields only" may need nuance. Confluence ac:* storage format macros have specific XML structure that must be tested in the actual Confluence instance. Medium complexity due to macro formatting.
#6 R4 Retrieval quality

Similarity score as article confidence gate

Confidence 88% Complexity Low
Description
Extract the top cosine similarity score from find_similar_tickets results. When the highest score falls below a configurable threshold (suggested: 0.75), inject a low-precedent mode instruction into the agent's context: "This ticket has low similarity to past resolved tickets (score: {score:.2f}). Flag any claims you cannot directly attribute to the provided sources." Expose the similarity score and mode in the SSE done event for optional UI use.
Cosine similarity score (already computed, currently unused) gates the agent into a low-precedent mode below the threshold.
Basis
direct snowflake.py:104–107 — similarity_score is computed via VECTOR_COSINE_SIMILARITY and returned in the result dict. agent.py:159–170 passes the entire result to the agent as raw JSON but never extracts similarity_score or uses it to condition behavior anywhere in the pipeline. The data is already there; nothing has to be added to Snowflake.
Rationale
Cosine similarity is a direct measure of how much historical precedent exists for this ticket type. A similarity score below 0.75 signals the agent is in low-precedent territory, where the model has fewer analogous resolved tickets to draw on — increasing hallucination risk on specific resolution steps. Injecting a low-precedent prompt instruction is cheaper than post-hoc verification and activates an existing signal that is currently ignored entirely.
Downsides
Threshold requires calibration against real data — 0.75 is a starting estimate. Low-precedent mode may over-hedge on some legitimately novel tickets where the agent can still produce accurate content. The similarity score in the current query is for ticket-to-ticket similarity, which proxies but does not directly measure article accuracy risk.
#7 H5 Human feedback loop

Structured micro-feedback form → coverage_records

Confidence 82% Complexity Medium
Description
Add a lightweight structured feedback step before the publish button activates. Five checkboxes: (1) root cause matches ticket, (2) resolution steps are reproducible, (3) missing context or sources, (4) article title is accurate, (5) draft needs significant editing. Optional 280-char free-text note. Store the feedback as structured columns alongside the coverage_record. After 50+ articles, the structured signal reveals which failure modes are most common — directly informing prompt iteration.
Basis
direct db.py:29–34 — coverage_records schema has no quality signal columns: only component, confluence_url, title, published_at. The UI publish button (index.html) is a single click with no checkpoint. No feedback mechanism exists anywhere in the pipeline — there is no way to record why a draft was accepted, edited, or rejected.

reasoned Structured checkboxes are aggregatable across articles; free-text is not. After N articles, SELECT failure_mode, COUNT(*) FROM quality_signals GROUP BY failure_mode directly reveals whether "missing context" or "inaccurate root cause" is the dominant failure — which maps to specific prompt rules to iterate on.
Rationale
The agent currently generates articles with no mechanism for the TEE to record why they accepted, edited, or discarded a draft. Structured pre-publish feedback is low-friction (5 clicks max) and turns each publish event into a labeled quality signal. Unlike regeneration (H2) or inline editing (H1), this idea closes the feedback loop at the data layer — enabling systematic prompt improvement based on observed failure patterns rather than intuition.
Downsides
Adds a step to every publish flow — needs careful UX design to avoid checkbox fatigue. Requires a schema migration to add quality signal columns to coverage_records (or a new quality_signals table). The signal only becomes useful after enough articles accumulate — low value until N ≈ 50.
#8 C1 Retrieval quality

CONCEPTS.md vocabulary constraint injection

Confidence 85% Complexity Low
Description
Pass the team's canonical vocabulary from CONCEPTS.md as a constraint block appended to SYSTEM_PROMPT. Instruct the model to prefer terms defined in CONCEPTS.md — e.g., use TEE not support engineer, KB Article not knowledge base document, Output Contract not output format. As CONCEPTS.md grows, the vocabulary constraint grows automatically with no code change.
Basis
direct CONCEPTS.md was added to the repo root this session with 7 defined domain terms (KB Article, TEE, Ticket, Output Contract, Done Event, Verdict Surface, Silent Context Tool). prompts.py SYSTEM_PROMPT step 4 instructs the agent to "inform accurate product terminology" but provides no canonical source. No vocabulary constraints or term definitions appear in prompts.py. The model defaults to training-distribution synonyms.
Rationale
When a TEE reads an article that calls them a "support engineer" or describes "knowledge base documents" instead of "KB Articles," it signals the tool doesn't understand internal vocabulary — reducing trust in the content. Injecting CONCEPTS.md terms as a constraint is cheap: read the file once at prompt assembly time in prompts.py, append a vocabulary block. No Snowflake call, no API overhead.
Downsides
SYSTEM_PROMPT token count grows with every CONCEPTS.md entry. Long-term, a dedicated vocabulary lookup tool is a cleaner approach; direct injection is fine while the vocabulary is small. The quality benefit is reasoned — LLMs in long prompts sometimes deprioritize low-frequency constraints buried at the end.
#9 E1 Human feedback loop

Section-level textarea editing before publish

Confidence 82% Complexity Low
Description
Add an Edit button next to each required H2 section in the preview panel. Clicking opens a modal with a plain <textarea> pre-populated with that section's raw HTML. On save, the textarea value replaces the corresponding section in currentDraft.draft_html via a string replace on the H2 anchor. No contenteditable — all edits write directly to the currentDraft object, which the existing publishDraft() reads unchanged.
Basis
direct H1 (contenteditable inline editing) was weakened in the original pass because publishDraft() reads from currentDraft.draft_html, not from the DOM — making in-DOM edits invisible to the publish path. A textarea modal that writes back to currentDraft.draft_html resolves this without changing the publish path. Prerequisite: A1 (mandatory named section schema) provides the predictable H2 anchors this relies on.
Rationale
TEEs currently cannot fix a specific wrong sentence without discarding the full draft. The data flow is clean: textarea value → currentDraft.draft_html → publishDraft(). Implementation: ~30 lines of JS and a simple modal. Unlike H1's contenteditable approach, there is no invisible-edit risk, no DOMPurify conflict, and no DOM mutation complexity.
Downsides
Depends on A1 (mandatory H2 sections) — without predictable section anchors, section targeting by regex is fragile. The textarea exposes raw Confluence storage format XML, which TEEs may find unfamiliar. Requires a note ("editing raw Confluence HTML") alongside the textarea. H2-based splitting may fragment sections that span multiple headings.
#10 U1 Prompt & synthesis

Explicit uncertainty-marker instructions in prompt

Confidence 80% Complexity Low
Description
Add a quality rule to SYSTEM_PROMPT: "For any resolution step or root cause claim you cannot directly attribute to the ticket conversation, Jira payload, or similar ticket data, wrap the claim in a Confluence ac:expand macro labeled 'Verify before publishing.' Do not add expand macros to claims you can directly cite." This makes the model's uncertainty visible inside the published article without the source-composition tracking required by P1.
Basis
direct prompts.py quality rule (line ~33): "if the root cause is not definitively confirmed, say so" — no output format specified. The model currently expresses uncertainty in prose, which TEEs must read carefully for hedging language. No ac:expand macro usage appears anywhere in the prompt or output format. Confluence ac:expand is valid Confluence storage format and renders as a collapsible section.
Rationale
A simpler, earlier-to-ship complement to P1. P1 requires computing source composition in Python and injecting it as prompt context; U1 requires only a prompt rule change — one edit to prompts.py. The two address different granularities: U1 marks claim-level uncertainty inside the article body; P1 marks source-level confidence at the article level via callout macros. U1 ships in one prompt edit and does not block or conflict with P1.
Downsides
Confluence ac:expand XML structure must be tested in the actual Confluence instance — LLMs frequently produce malformed Confluence storage format on complex macros. "Cannot directly attribute" is a fuzzy instruction that models apply inconsistently without example thresholds. Once P1 ships, U1's expand macros may become redundant and need a cleanup prompt edit.
#11 V1 Retrieval quality

Component coverage & gap context pre-injection

Confidence 75% Complexity Medium
Description
Before the agent loop starts, query both coverage_records and gap_records for the current ticket's primary_product_component. Inject the result into the agent's first message: "Component {X} currently has {N} KB articles published (most recent: '{Title}') and {M} unresolved escalation patterns tracked. Avoid duplicating angles already covered; prioritize gaps with no existing article." This gives the agent the full component landscape — what's been written and what's still missing — before drafting.
Basis
direct db.py:20–34 defines both gap_records (unsolved ticket patterns per component) and coverage_records (published articles per component). db.py:40 — get_coverage() returns component-level data. agent.py:159 — component_holder[0] is populated from fetch_ticket result at tool call time, not pre-loop. Neither table is referenced in run_kb_agent(). Both data sources are surfaced only in the UI Coverage tab, not in the generation loop.
Rationale
The coverage DB tracks what's been written; the gap DB tracks what's still needed. Feeding both to the agent before drafting means the article is oriented toward genuine gaps rather than duplicating existing coverage. The data sources are already there — this is a SQLite read and a few extra tokens in the first message, not new infrastructure.
Downsides
primary_product_component is known only after fetch_ticket resolves, so true pre-injection requires a two-phase message setup (fetch first, then inject, then start loop). Does not apply to raw-text mode. Coverage DB only contains articles published through this tool — manually published articles are invisible. Sparse at low article volumes.
